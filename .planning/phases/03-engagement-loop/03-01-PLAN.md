**1. app/api/archetype/submit/route.ts** — POST endpoint:
- Import db, users, archetypes from schema, eq from drizzle-orm
- Get userId from headers (x-user-id, set by middleware)
- Parse body: { archetype: string } (one of 'builder', 'analyst', 'explorer', 'collaborator')
- Validate archetype name is one of the 4 valid values
- Query archetypes table to get the archetype ID by name
- Update users table: set archetypeId where id = userId
- Return { success: true, archetype: { name, color, iconKey } }
- Error handling: 400 for invalid archetype, 404 for archetype not found, 401 for no user

**2. middleware.ts** — Add archetype quiz redirect:
- After verifying JWT and extracting user info, check if the user is a learner (role === 'learner') and has no archetypeId
- If learner without archetype AND the request path is NOT /archetype-quiz AND NOT an API route AND NOT a static asset:
  - Redirect to /archetype-quiz
- If learner WITH archetype AND path IS /archetype-quiz:
  - Redirect to / (they already have an archetype)
- Keep existing behavior for non-learner roles (mentors/admins skip quiz)
- The archetypeId check can use the x-user-archetype header or a DB lookup. Since the JWT might not contain archetype info, the simplest approach is to add archetypeId to the JWT payload.
  - ALTERNATIVE: Simpler — just check if x-user-archetype header exists (if the middleware already sets it from JWT). Check what the current middleware does.
  - If the JWT doesn't contain archetype: the middleware can do a quick DB check. OR: after submitting archetype, refresh the JWT cookie with the archetypeId included.
  - DECISION: After archetype submit, reissue the JWT with archetypeId in the payload. Middleware reads it from JWT claims. Update the submit endpoint to reissue the cookie.
